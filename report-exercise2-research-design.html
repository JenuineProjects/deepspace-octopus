Exercise 2: DeepSpace Research & Design Phase — Tier 2 Team Orchestration

Date: March 3, 2026 | Framework: Octopus Multi-Agent Orchestration OS | Model: Claude Opus 4.6

1. Executive Summary

This exercise demonstrated the Parallel Fan-Out orchestration pattern with a Tier 2 Agent Team. The Purple Manager dispatched two specialist agents — a Yellow Researcher and a Red-Orange Designer — in parallel. The Researcher conducted market intelligence on productivity apps; the Designer produced a system architecture and UI blueprint. The Manager then synthesized both outputs into a single Phase 1 Blueprint for human approval.

Both agents completed successfully. The Manager's integration produced a comprehensive blueprint covering architecture, UI, data models, and research-backed design decisions — ready for the Build Phase.

2. How Octopus Was Used

The Octopus framework structured this exercise through:

  • Tier 2 Agent Team: Two agents with explicit contracts (domain, task, output format, constraints) operating as teammates under Manager coordination.
  • Parallel Fan-Out Pattern: Both agents were dispatched simultaneously. The Designer was instructed to consume the Researcher's output if available, or proceed independently.
  • Lateral Trigger Protocol: The Researcher was instructed to flag standout UI patterns for immediate cross-agent communication, without waiting for Manager integration.
  • Output Contracts: Each agent had a strict output contract — specific files, specific directories, specific JSON/Markdown formats, and a status.json completion signal.
  • Manager Synthesis: Rather than passing raw agent outputs to the human, the Manager synthesized both into a single coherent blueprint — adding analysis, recommendations, and open questions.

4.2 How Agents Integrated

  • Parallel dispatch: Both agents were launched simultaneously via Claude Code's Agent tool. This is the Octopus "Fan-Out" pattern — independent tasks run concurrently, Manager collects results.
  • Attempted cross-agent communication: The Designer was instructed to read the Researcher's market-brief.json if available. The Designer checked 3 times but the file wasn't ready yet (the Researcher was still running). The Designer proceeded independently.
  • Lateral triggers (partial): The Researcher produced 5 lateral triggers flagged for the Designer. Because the agents ran concurrently and the Designer couldn't wait, these triggers were integrated by the Manager during synthesis rather than consumed directly by the Designer.
  • Manager as integrator: The Manager read all outputs, cross-referenced the Researcher's findings with the Designer's architecture, and produced a unified blueprint that incorporated both perspectives.
Observation: Timing Gap in Parallel Execution

The lateral trigger protocol assumed the Researcher would finish before the Designer needed the data. In practice, both agents ran concurrently and the Designer couldn't consume the market brief. This is a known trade-off of parallel fan-out vs. sequential pipeline. The Manager compensated by integrating the lateral triggers during synthesis. In a production system, a message queue or event-driven trigger would enable true real-time cross-agent communication.

8. How Well It Worked

8.1 Strengths

  • Parallel execution saved time: Both agents ran concurrently. In a sequential pipeline, the Designer would have waited for the Researcher. The parallel approach cut wall-clock time roughly in half.
  • Deep research with real data: The Researcher used web search to gather current pricing, user counts, complaints, and reviews for all 8 apps. This isn't synthetic — it's grounded in real market data.
  • Research-backed architecture: The Manager's synthesis cross-referenced the Researcher's psychological principles with the Designer's technical decisions, creating justifications for every major design choice.
  • Comprehensive design contract: The UI schema is detailed enough to hand directly to a developer — screen components, state transitions, data models, and design tokens are all specified.
  • Manager discipline held: The Manager synthesized and added value (cross-referencing, recommendations, open questions) rather than simply concatenating outputs.

8.2 Limitations

  • Lateral triggers weren't consumed in real-time: The cross-agent communication protocol (Researcher flags patterns → Designer reads them) didn't work because both agents ran concurrently. The Manager compensated during synthesis, but the Designer's architecture wasn't directly shaped by the research findings.
  • Designer worked without research: Because the market brief wasn't available, the Designer proceeded from the master intent alone. The architecture is sound but might have been different (potentially better) with the competitive analysis available.
  • No validation loop: Unlike Exercise 1's review loop, this exercise had no quality gate between agents. The Manager accepted both outputs without sending either back for revision.
  • Research depth vs. breadth: Web search results vary in quality. Some competitive data (exact user counts, specific complaint details) may be approximations rather than verified figures.

8.4 Octopus Framework Assessment

The Octopus framework provided clear structure for a multi-agent research and design phase. Key observations:

  1. Agent contracts work: Giving each agent a defined domain, output format, and scope constraint produced focused, high-quality outputs. Neither agent strayed outside its role.
  2. Parallel fan-out is fast but lossy: Running agents in parallel saves time but sacrifices the ability for later agents to build on earlier results. For research → design workflows, a sequential pipeline (Exercise 1 style) would preserve information flow.
  3. Manager synthesis adds real value: The integrated blueprint is significantly more useful than the raw agent outputs. Cross-referencing research findings with architectural decisions created a coherent narrative that neither agent could produce alone.
  4. Filesystem-as-truth scales: The structured directory tree with status files, output contracts, and state tracking made it easy to monitor progress and verify completion.
  5. The framework is composable: Exercise 1 used Review Loop. Exercise 2 used Fan-Out. Both used the same agent contracts, status protocols, and filesystem conventions. The patterns are modular building blocks.

9. Recommendations for Future Exercises

  • Use sequential pipeline when agents have dependencies: If the Designer needs the Researcher's output, run them sequentially. Reserve parallel fan-out for truly independent tasks.
  • Add a review gate to design outputs: A dedicated review step (like Exercise 1's @reviewer) would catch gaps in the architecture before the Build Phase.
  • Implement event-driven lateral triggers: Instead of filesystem polling, a message queue or shared event log would enable real-time cross-agent communication during parallel execution.
  • Combine patterns: The ideal workflow would be: Research (sequential) → Design (sequential) → Review Loop (iterative) → Build (with review loop) → Market (parallel fan-out). This uses all four Octopus orchestration modes.
